VideoRecorderView_Refactored.swift Audit
==========================================
Date: 2026-04-13

Files Reviewed:
- VideoRecorderView_Refactored.swift (all sections: view, trimmer, network monitor, player layer)
- VideoUploadService.swift (referenced dependency)
- CameraViewModel.swift (for comparison with DirectCameraRecorderView patterns)


BUGS
====

1. Orientation lock never restored on some discard paths
   File: VideoRecorderView_Refactored.swift:123-131, 151-160, 179-187

   The pendingDismissAction closures for the trimmer cancel and play
   result discard paths set showingTrimmer = false and clean up video
   URLs, but do NOT restore the orientation lock:

     PlayerPathAppDelegate.orientationLock = .allButUpsideDown

   This is only restored in the onSave success paths (lines 118, 146)
   and in cleanupAndDismiss (line 400). But pendingDismissAction at
   lines 123-131 and 151-160 set their own closures that execute BEFORE
   cleanupAndDismiss runs (line 80-82 in the confirmation dialog). So
   cleanupAndDismiss does restore it -- but if the user dismisses via
   those paths, the flow is: pendingDismissAction() -> cleanupAndDismiss()
   which does call line 400. This is actually OK on closer inspection.

   HOWEVER: The trimmer cancel path (lines 179-187) does NOT call
   lockPortrait() beforehand, so orientation was never locked for that
   path. But the trimmer onSave and onSkip DO call lockPortrait() (lines
   170, 175). So if the user trims, then discards from the play result
   screen, the orientation was locked to portrait and gets restored. If
   the user cancels from the trimmer itself, orientation was never locked
   so no restore is needed. This is correct but fragile -- any future
   change to the flow could break it.


2. Duplicate NetworkMonitor when ConnectivityMonitor.shared exists
   File: VideoRecorderView_Refactored.swift:435-487

   This file defines its own NetworkMonitor class using NWPathMonitor,
   but the project already has ConnectivityMonitor.shared (in
   Services/ConnectivityMonitor.swift) that does the same thing. This
   is redundant code that could drift out of sync.

   The local NetworkMonitor also has dead code in the pathUpdateHandler
   (lines 453-458): it computes interfaceTypeName but assigns it to a
   discarded _ variable, and the else branch is empty.

   Fix: Replace with ConnectivityMonitor.shared.isConnected.


3. PhotosPicker auto-dismiss race condition
   File: VideoRecorderView_Refactored.swift:66-69

   When showingPhotoPicker becomes false:
     if !isShowing && selectedVideoItem == nil && recordedVideoURL == nil && !showingTrimmer {
         dismiss()
     }

   This auto-dismisses when the photo picker closes without a selection.
   But handleSelectedVideo runs async (line 266), so there's a brief
   window where:
   - User selects a video
   - Picker closes (showingPhotoPicker = false)
   - onChange fires, sees selectedVideoItem is set BUT recordedVideoURL
     is still nil (async processing hasn't finished)
   - showingTrimmer is still false

   The selectedVideoItem != nil check saves this particular case. But
   if the processSelectedVideo call fails (returns .failure), then
   selectedVideoItem is non-nil but recordedVideoURL stays nil and
   showingTrimmer stays false. The user is stuck on a black screen with
   no way to dismiss except swiping.

   Fix: On .failure, explicitly dismiss or show an error alert.


4. PreUploadTrimmerView slider bounds can produce invalid range
   File: VideoRecorderView_Refactored.swift:706-707

   The slider bounds are computed as:
     let startSliderMax = max(0.1, endTime - 0.5)
     let endSliderMin = min(duration - 0.1, startTime + 0.5)

   If the video is exactly 0.6s long (the minimum for showing sliders):
   - startTime=0, endTime=0.6
   - startSliderMax = max(0.1, 0.6-0.5) = max(0.1, 0.1) = 0.1
   - endSliderMin = min(0.6-0.1, 0+0.5) = min(0.5, 0.5) = 0.5
   - Start slider range: 0...0.1, End slider range: 0.5...0.6

   This works, but if the user drags startTime to 0.1:
   - endSliderMin = min(0.5, 0.1+0.5) = 0.5
   - End slider range: 0.5...0.6 (still OK)

   If user drags endTime to 0.5:
   - startSliderMax = max(0.1, 0.5-0.5) = max(0.1, 0) = 0.1
   - Still OK. But if endTime reaches below 0.5, startSliderMax goes
     to 0 (clamped to 0.1), and the ranges stay valid.

   The real issue: when duration is between 0.5 and 0.6 (possible with
   float rounding), the guard at line 704 (duration >= 0.6) excludes it,
   so this edge case is moot. The bounds are OK.


5. isPlaying state accessed from non-main-actor closure
   File: VideoRecorderView_Refactored.swift:873, 887

   The periodic time observer closure (line 867) and the notification
   observer closure (line 884) run on .main queue, but they read and
   write @State properties (isPlaying, currentTime) directly. Since
   PreUploadTrimmerView is a struct (not @MainActor), these closures
   capture self's state bindings. In practice this works because the
   queue is .main, but it technically accesses SwiftUI @State from
   a non-SwiftUI context. Under strict concurrency this will warn.


POTENTIAL ISSUES
================

6. VideoUploadService.isProcessingVideo is never observed in the UI
   File: VideoRecorderView_Refactored.swift:33

   @StateObject private var uploadService = VideoUploadService() is
   created but uploadService.isProcessingVideo is never checked in the
   view body. The user sees no loading indicator while a video is being
   imported from Photos. For large 4K videos, this could take several
   seconds with no feedback.


7. saveVideoWithResult has redundant MainActor.run calls
   File: VideoRecorderView_Refactored.swift:325-371

   The entire view is @MainActor, and saveTask is created as Task { }
   which inherits main actor isolation. The await MainActor.run { }
   calls inside are redundant. Not a bug, just unnecessary.


8. AVPlayerLayerView is private but could be shared
   File: VideoRecorderView_Refactored.swift:506-531

   AVPlayerLayerView is defined as private, but an identical pattern
   exists in ModernCameraView (CameraPreviewLayer.PreviewView). These
   could share a common UIView subclass, though the cost of duplication
   is low.


9. PreUploadTrimmerView player not muted for imported videos
   File: VideoRecorderView_Refactored.swift:858-859

   The trimmer player is created with isMuted = true (line 859). This
   is correct for camera recordings but unexpected for imported videos
   from Photos -- the user might want to hear audio to find the right
   trim point. The DirectCameraRecorderView trimmer (same component)
   also mutes, which makes more sense for fresh recordings.


10. Combine import unused
    File: VideoRecorderView_Refactored.swift:18

    Combine is imported but no Combine types are used in this file.
    The @Published properties are in NetworkMonitor and VideoUploadService,
    but those use ObservableObject which imports Combine implicitly.


11. Multiple redundant imports
    File: VideoRecorderView_Refactored.swift:8-19

    Several imports appear unnecessary for this file:
    - AVKit: not used directly (AVPlayer is in AVFoundation)
    - CoreMedia: CMTime is available via AVFoundation
    - UIKit: available implicitly via SwiftUI on iOS
    - Combine: not used directly
    - Network: only used by the local NetworkMonitor class

    If NetworkMonitor is replaced with ConnectivityMonitor.shared (#2),
    then Network import can also be removed.


12. GuidelineItem view appears unused
    File: VideoRecorderView_Refactored.swift:408-430

    GuidelineItem is defined as a public struct but is never referenced
    anywhere in this file. It may be used elsewhere, but if not, it's
    dead code.


MINOR / CLEANUP
===============

13. Dead code in NetworkMonitor.pathUpdateHandler
    File: VideoRecorderView_Refactored.swift:453-458

    The if/else branches after setting isConnected and connectionType
    do nothing useful:
    - The if branch computes interfaceTypeName() and discards it (_ =)
    - The else branch is completely empty

    This looks like leftover debug logging that was removed.


14. No accessibility for trimmer sliders
    File: VideoRecorderView_Refactored.swift:714, 726

    The start and end trim sliders have no accessibility labels. Screen
    reader users would hear generic "slider" with no indication of what
    each controls. Add:

      .accessibilityLabel("Start time")
      .accessibilityLabel("End time")
