  CoachDetailView, and a preview. Consider extracting CoachDetailView
  and AddCoachView into separate files for maintainability.

[LOW] Unused `athleteID` property (line 31)

  Should use `.textInputAutocapitalization(.never)` instead.

[LOW] `deleteCoach` wraps `Task` inside `withAnimation` (line 285-320)

  the animation context anyway. The animation block should only wrap
  the synchronous `modelContext.delete(coach)` call.

  Both `CoachesView.deleteCoach` (line 277) and `CoachDetailView.deleteCoach`

  The entire mailto URL is percent-encoded including the scheme.
  `addingPercentEncoding(withAllowedCharacters: .urlQueryAllowed)` on

  Better to encode subject and body separately.

[INFO] TODO at line 856
  "TODO: Implement revoke access" in CoachDetailView — shared folder
  revoke button is a no-op.

9. ComprehensiveAuthManager.swift (1074 lines)
───────────────────────────────────────────────
Purpose: Central auth manager — Firebase Auth (email/password + Apple
         Sign In), user profile, role management, subscription tier
         sync, biometric auth, account deletion.

ISSUES:

[HIGH] Uses `import Combine` and `AnyCancellable` for StoreKit sync (line 86)
  CLAUDE.md states: "Avoid using the Combine framework and instead prefer
  to use Swift's async and await versions of APIs instead." The two
  `.sink` subscribers on StoreKitManager.$currentTier and
  .$currentCoachTier could be replaced with async `for await` on
  the respective AsyncSequences, but this requires StoreKitManager
  to expose AsyncStream properties. This is a significant refactor.

[MEDIUM] Excessive `#if DEBUG print()` statements (~40+ occurrences)
  The file has extensive debug logging via `print()`. The project
  convention is to use `Logger` (os.log). While print() is fine for
  development, the volume makes the file harder to read. Consider
  consolidating these into `authLog.debug()` calls.

[MEDIUM] Large file (1074 lines) with mixed responsibilities
  Handles authentication, profile management, subscription sync,
  local SwiftData sync, and account deletion. Consider extracting:
  - Subscription tier sync into a dedicated service
  - Profile loading/retry logic into a helper
  - Account deletion into a separate service

[LOW] `loadUserProfile` and `loadUserProfileWithRetry` have duplicated
  profile-processing logic (lines 540-623 vs 646-734)
  Both methods decode the profile, check roles, apply tier overrides.
  The retry variant should call the single-shot variant internally
  rather than duplicating the processing.

[LOW] `deinit` removes auth listener but singleton-like usage means it
  rarely fires. Class is injected via `.environmentObject()` so it
  lives as long as the root view. Not harmful.

[LOW] `hasAthleteTierOverride` marked "NOT persisted to UserDefaults
  (security: tamper-resistant)" (line 63) but `clearPersistedUserDefaults`
  removes a `hasAthleteTierOverride` key (line 842). The key removal
  is harmless (no-op if it doesn't exist) but inconsistent with the
  comment.

[INFO] Well-documented race condition handling
  Comments throughout explain why operations are ordered the way they
  are (e.g., loading profile BEFORE setting isSignedIn). This is
  good defensive documentation.

[INFO] Good GDPR deletion flow (lines 924-1017)
  Step-by-step deletion with continue-on-failure semantics ensures
  partial deletion is better than no deletion.

VERDICT: The central nervous system of the app. Functional and well-
         documented. Main concerns are file size, Combine usage vs
         project conventions, and duplicated profile-processing logic.

CROSS-CUTTING THEMES:

1. Unnecessary `import Combine` — Present in 6 of 9 files. None use
   Combine-specific APIs (publishers, subscribers, operators). Can be
   safely removed from all view files. ComprehensiveAuthManager is the
   only file that actually uses Combine (AnyCancellable, .sink).

   ComprehensiveAuthManager (1074) are all over 900 lines with multiple
   embedded types. These should be split for maintainability.

   CoachFolderViewModel use ObservableObject. ComprehensiveAuthManager
   uses ObservableObject with Combine publishers. Project convention
   prefers @Observable.

4. Duplicated code:
   - `deleteCoach` logic in CoachesView and CoachDetailView
   - `formatTimestamp` in NoteCardView and VideoInfoTabView
   - Profile-processing logic in loadUserProfile and loadUserProfileWithRetry

5. Import hygiene — Several files import FirebaseAuth or FirebaseFirestore
   without using them directly (access is through managers/services).
