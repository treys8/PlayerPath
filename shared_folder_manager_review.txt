SharedFolderManager.swift Audit
==================================
Date: 2026-04-13

Files Reviewed:
- SharedFolderManager.swift


BUGS
====

1. notifyCoachAccessRevoked resolves wrong user's display name
   File: SharedFolderManager.swift:193-199

   When resolving the athlete's name for the revocation notification:

     let name = Auth.auth().currentUser?.displayName
         ?? profile.email.components(separatedBy: "@").first
         ?? profile.email

   Auth.auth().currentUser is the CURRENTLY SIGNED-IN user (the athlete
   performing the revocation). But profile is fetched for athleteID.
   If the signed-in user's UID differs from athleteID (e.g., a parent
   account managing multiple athletes), this returns the parent's
   display name, not the athlete's name.

   The code fetches the athlete's UserProfile but then ignores it and
   falls back to Auth.auth().currentUser?.displayName. It should use
   fields from the fetched profile instead.

   Fix: Use profile.email-derived name consistently, or add a
   displayName field to UserProfile. For example:

     let name = profile.email.components(separatedBy: "@").first
         ?? profile.email


2. deleteFolder doesn't notify coaches with folder name context
   File: SharedFolderManager.swift:233-251

   In deleteFolder, the coach permission revocation (line 235) calls
   firestore.removeCoachFromFolder WITHOUT folderName, coachEmail, or
   athleteID parameters:

     try await firestore.removeCoachFromFolder(folderID: folderID, coachID: coachID)

   But removeCoachAccess (line 155) passes all of these. The 2-param
   overload of removeCoachFromFolder may not record the revocation
   event with full context, meaning the coach may not see a meaningful
   notification about which folder was removed.

   The notification at line 244 also passes coachEmail: "" which means
   the notification service can't do email-based lookups if needed.


3. Listener 50-folder hard limit with no pagination
   File: SharedFolderManager.swift:309, 366

   Both athlete and coach folder listeners use .limit(to: 50). If an
   athlete or coach has more than 50 folders, the extras are silently
   dropped. There's no UI indication that the list is truncated, and
   no way to load more.

   Impact: Academy-tier coaches could potentially have 50+ folders
   (one per athlete, unlimited athletes). They'd silently lose
   visibility of older folders.

   Fix: Either increase the limit, add pagination, or show a "showing
   50 of N folders" indicator.


POTENTIAL ISSUES
================

4. uploadVideo fetches folder twice on success
   File: SharedFolderManager.swift:550 and 703

   The folder is fetched at line 550 to verify permissions, then
   fetched again at line 703 to get coach IDs for notifications. The
   second fetch happens after the upload completes, so if coaches were
   added/removed during upload, the notification targets the updated
   list. This is arguably correct behavior, but it's an extra Firestore
   read that could be avoided by caching the first fetch.


5. Compression uses original URL for duration extraction
   File: SharedFolderManager.swift:628

   After compression, the code extracts duration from videoURL (the
   original), not uploadURL (the compressed copy). This is correct
   since compression shouldn't change duration, and the original is
   guaranteed to exist. Just noting the pattern.


6. acceptInvitation swallows loadCoachFolders error
   File: SharedFolderManager.swift:486

     try? await loadCoachFolders(coachID: coachID)

   If the folder refresh fails after accepting, the invitation is
   accepted server-side but the UI doesn't update. The user might
   think acceptance failed. The try? silently drops the error.

   Impact: Low -- the listener would pick up the change on next
   snapshot, or pull-to-refresh would fix it. But a transient network
   error here means the user sees no immediate feedback.


7. createFolder tier check is redundant
   File: SharedFolderManager.swift:57

     guard hasCoachingAccess, StoreKitManager.shared.currentTier >= .pro else {

   The caller passes hasCoachingAccess, and then the code ALSO checks
   StoreKitManager. If the intent is "don't trust the caller", then
   hasCoachingAccess is redundant. If both are needed, the logic is:
   both must be true, which means if StoreKit says Pro but the caller
   says false, it fails. This seems overly conservative -- the StoreKit
   check alone should suffice.


8. isLoading set in listeners but not reset on listener stop
   File: SharedFolderManager.swift:303, 349-352

   startAthleteFoldersListener sets isLoading = true (line 303). If
   stopAthleteFoldersListener is called before the first snapshot
   arrives, isLoading remains true. The clearAllData method (line 418)
   does reset it, but stopAthleteFoldersListener alone doesn't.

   Impact: If a view calls stop without clear, it could show a
   perpetual loading state.


9. Listener snapshot closures use [weak self] but manager is a singleton
   File: SharedFolderManager.swift:310, 367

   The snapshot listeners capture [weak self], but SharedFolderManager
   is a singleton (static let shared). It will never be deallocated, so
   weak self will never be nil. The [weak self] is harmless but
   misleading -- it suggests the manager could go away, which it can't.

   Not a bug, just unnecessary ceremony.


10. deleteVideo doesn't decrement folder videoCount
    File: SharedFolderManager.swift:724-742

    When a video is deleted, the Storage file and Firestore doc are
    removed, but the folder's videoCount field is not decremented.
    Compare with markVideoCompleted (line 681) which atomically
    increments the count. The asymmetry means videoCount drifts
    upward over time as videos are deleted.

    The actual count may be maintained by deleteVideo in
    FirestoreManager, but this should be verified. If the Firestore
    deleteVideo function doesn't decrement videoCount, the dashboard
    and folder cards will show inflated video counts.


MINOR / CLEANUP
===============

11. Unused imports
    File: SharedFolderManager.swift:10-11

    AVFoundation and CoreMedia are imported but only used indirectly
    (AVURLAsset at line 628). The actual AVFoundation usage is minimal
    and could be isolated. Foundation alone may suffice if the duration
    extraction were moved to a helper.


12. Empty line after invitation creation
    File: SharedFolderManager.swift:126

    Blank line between invitation creation and the notification code.
    Minor formatting.


13. leaveFolder doesn't verify coach actually has access first
    File: SharedFolderManager.swift:513-518

    leaveFolder calls removeCoachFromFolder without checking if the
    coach is actually in the folder's sharedWithCoachIDs. If called
    with the wrong coachID, the Firestore operation would either
    no-op or error. A local check would fail faster.
