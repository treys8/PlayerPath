AppUpdateManager.swift Review
=============================

Overall Assessment
------------------
Compact, focused service that checks Firestore appConfig/current for forced updates
and What's New content. Clean singleton with two @Published properties driving
ForceUpdateView and WhatsNewView in MainAppView. Non-fatal error handling on config
fetch failure. Semantic version comparison is correct. Few minor issues below.

Issues
------

[AU1] LOW â€” latestVersion and minimumVersion are dead state
  Lines 24-25: Both `private(set) var minimumVersion` and `private(set) var
  latestVersion` are written from Firestore data (lines 43-45) but never read by
  any consumer. ForceUpdateView receives `updateURL` but doesn't use the other two.
  These properties store data that nobody accesses.

[AU2] LOW â€” Uses ObservableObject/@Published instead of @Observable
  Line 17: The class conforms to `ObservableObject` with `@Published` properties,
  while the project convention (per CLAUDE.md) is `@Observable`. MainAppView uses
  `@ObservedObject` to observe it. Migration to `@Observable` would align with the
  project convention and eliminate the Combine import.

[AU3] LOW â€” import Combine needed only because of ObservableObject
  Line 10: The Combine import exists solely to support `ObservableObject` and
  `@Published`. If migrated to `@Observable` per [AU2], this import becomes
  unnecessary.

[AU4] LOW â€” isVersion silently drops non-numeric components
  Lines 90-91: `compactMap { Int($0) }` silently discards non-numeric version
  segments (e.g., "3.24.26-beta" becomes [3, 24]). App Store versions are always
  numeric so this is safe in practice, but a malformed Firestore minimumVersion
  string could produce an incorrect comparison.

[AU5] LOW â€” Bundle.main.appVersion read three times across two methods
  Lines 48, 60, 76: The app version is fetched from the bundle info dictionary
  three separate times â€” twice in checkOnLaunch and once in markWhatsNewSeen.
  Could be captured once as a local let in checkOnLaunch and stored as a property
  for markWhatsNewSeen.

Positive Patterns
-----------------
- Non-fatal config fetch: catch block on line 68-71 logs and continues, never
  blocking app launch for a Firestore failure
- Force-update gate: requiresUpdate is checked first in MainAppView body,
  preventing any app interaction until the user updates
- Early return on force update: line 52 skips What's New display when an update
  is required, avoiding confusing double-modal scenarios
- UserDefaults persistence: What's New seen state persists across app launches
  via lastSeenWhatsNewVersion key
- Semantic version comparison: isVersion correctly handles versions with different
  component counts (e.g., "3.24" vs "3.24.1") by zero-padding
- Single Firestore read: One getDocument() call fetches both force-update and
  What's New data, minimizing Firestore reads
