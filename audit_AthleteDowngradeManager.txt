AthleteDowngradeManager.swift Review
=====================================
File: PlayerPath/Services/AthleteDowngradeManager.swift (86 lines)
Date: 2026-04-21

Overall Assessment
------------------
Small, focused service that detects when an athlete has shared coach folders but lost
Pro tier. Sets a lapsed state and notifies coaches via ActivityNotificationService.
Non-destructive by design — doesn't auto-revoke access, so re-subscribing restores
everything. Clean transition detection to avoid duplicate notifications. A few issues
with dead observable state and notification reliability.

Issues
------

[AD1] MEDIUM — state property is @Observable but never observed by any view
  Line 29: `private(set) var state: State` is published via @Observable, but no view
  ever reads it. MainTabView.swift:33 holds a reference to the manager and calls
  evaluate(tier:) on lines 158 and 166, but never checks .state. The state exists
  purely for internal transition detection (previousState != .lapsed to avoid
  duplicate notifications). A simple private Bool flag like `hasNotifiedCoaches`
  would serve the same purpose without the @Observable overhead.

[AD2] LOW — Coach notification is a one-shot with no retry
  Lines 49-53: When state first transitions to .lapsed, notifyCoachesOfLapsedStatus()
  fires once. If the notification fails internally (network issue, Firestore timeout),
  state is already .lapsed, so subsequent evaluate() calls won't re-trigger the
  notification (previousState == .lapsed check fails). Coaches may never learn
  about the lapse.

[AD3] LOW — Direct FirebaseAuth coupling for athlete identity
  Lines 66-67: The manager reaches into Auth.auth().currentUser for displayName and
  uid. Passing these as parameters to evaluate() or storing them from the caller
  would reduce FirebaseAuth coupling and improve testability.

[AD4] LOW — folder.id ?? "" passes empty string for missing Firestore IDs
  Line 76: If a SharedFolder has no Firestore ID, an empty string is passed to
  postAccessLapsedNotification as the folderID. This would create a notification
  with an invalid folder reference. Should skip folders without IDs.

Positive Patterns
-----------------
- Non-destructive design: Doesn't auto-revoke coach access on downgrade; folders
  and permissions persist so re-subscribing restores everything seamlessly
- Transition detection: Tracks previousState to fire coach notifications only once
  when first transitioning to lapsed, not on every evaluate() call
- Coach dedup: Uses notifiedCoachIDs Set to avoid sending duplicate notifications
  when the same coach has access to multiple folders
- Reactive evaluation: Called on both app launch (MainTabView.swift:158) and tier
  changes (.onChange of authManager.currentTier, line 166)
- Appropriate logging: Logs state transitions at info level, avoids noisy logging
  when state is already correct

Usage
-----
Used in 1 file:
  - MainTabView.swift: Holds .shared reference, calls evaluate(tier:) on appear
    and on tier changes. Never reads .state.

Summary: 1 MEDIUM, 3 LOW. Well-designed non-destructive downgrade handler. The main
issue is [AD1] — the @Observable state is never observed by any view, making the
observation machinery dead weight. [AD2] the one-shot notification with no retry is
a minor reliability gap.
