VideoPlayerView.swift Audit
=============================
Date: 2026-04-13

Files Reviewed:
- VideoPlayerView.swift


BUGS
====

1. saveToPhotos uses DispatchQueue.main instead of @MainActor
   File: VideoPlayerView.swift:293-315

   saveToPhotos() uses PHPhotoLibrary.requestAuthorization and
   performChanges with completion handlers that dispatch back to main
   via DispatchQueue.main.async. But VideoPlayerView is a struct (not
   explicitly @MainActor), and these closures capture self to mutate
   @State properties (isSavingToPhotos, saveErrorMessage, etc.).

   Under strict concurrency this is technically invalid -- @State
   mutations should happen from the SwiftUI view's actor context, not
   raw DispatchQueue.main. In practice this works on current iOS, but
   it will produce warnings under strict concurrency checking and could
   break in future Swift versions.

   Fix: Convert to async/await:

     private func saveToPhotos() {
         guard !isSavingToPhotos else { return }
         isSavingToPhotos = true
         let videoURL = clip.resolvedFileURL
         guard FileManager.default.fileExists(atPath: clip.resolvedFilePath) else {
             isSavingToPhotos = false
             saveErrorMessage = "Video file not found"
             return
         }
         Task {
             let status = await PHPhotoLibrary.requestAuthorization(for: .addOnly)
             guard status == .authorized || status == .limited else {
                 isSavingToPhotos = false
                 saveErrorMessage = "Photo library access denied."
                 return
             }
             do {
                 try await PHPhotoLibrary.shared().performChanges {
                     PHAssetChangeRequest.creationRequestForAssetFromVideo(atFileURL: videoURL)
                 }
                 Haptics.success()
                 showingSaveSuccess = true
             } catch {
                 saveErrorMessage = error.localizedDescription
             }
             isSavingToPhotos = false
         }
     }


2. ShareLink uses clip.resolvedFileURL which may not exist
   File: VideoPlayerView.swift:193-195

   ShareLink(item: clip.resolvedFileURL) is always present in the menu
   regardless of whether the local file exists. If the file was deleted
   or only exists in the cloud, the share will fail silently or produce
   a broken share sheet. The "Save to Photos" button has a file-exists
   check, but ShareLink does not.

   Fix: Conditionally show ShareLink only when the file exists, or
   wrap it with a check.


3. Cloud download overwrites filePath without checking for conflicts
   File: VideoPlayerView.swift:419

   After downloading from cloud:
     clip.filePath = VideoClip.toRelativePath(destinationPath)

   This mutates the SwiftData model's filePath. If another part of the
   app is also writing to the clip (e.g., SyncCoordinator), this could
   cause a conflict. The save is done via ErrorHandlerService.saveContext
   which should handle it, but there's no version check or merge logic.


4. onDisappear resets hasAppeared, causing re-setup on reappear
   File: VideoPlayerView.swift:234-243

   onDisappear sets hasAppeared = false and tears down the player
   entirely. If the view reappears (e.g., coming back from a sheet),
   onAppear will re-run setupPlayer from scratch. This means every
   sheet dismiss (play result editor, game linker, share to folder)
   triggers a full player teardown and re-setup, causing a visible
   loading flash.

   The onChange(of: clip.version) handler at line 229 already handles
   the re-trim case. The onDisappear teardown is too aggressive.

   Fix: Only tear down the player when truly leaving the view, not
   on sheet presentations. One approach is to check if the view is
   still in the navigation hierarchy before tearing down, or use a
   lighter cleanup that just pauses without destroying.


POTENTIAL ISSUES
================

5. setupPlayer redundant MainActor.run calls
   File: VideoPlayerView.swift:324-331, 339-343, etc.

   setupPlayer() is called from Task { await setupPlayer() } which
   inherits main actor isolation from the view context. The
   await MainActor.run { } blocks inside are redundant since the
   function is already running on the main actor.

   However: the function is not explicitly marked @MainActor, and it
   calls findVideoURL() which does async work. After each await, Swift
   may hop off the main actor in some cases. The MainActor.run calls
   are defensive and correct here. Not a bug, just noting the pattern.


6. Download progress polling loop
   File: VideoPlayerView.swift:404-411

   Download progress is polled every 200ms in a tight loop:

     while !Task.isCancelled {
         if let progress = cloudManager.downloadProgress[clipId] {
             downloadProgress = progress
         }
         try? await Task.sleep(for: .milliseconds(200))
     }

   This works but is wasteful. A better approach would be to use
   AsyncStream or a callback from VideoCloudManager to push progress
   updates. The polling also continues even if the download completes
   between checks (though progressTask.cancel() at line 416 stops it).


7. fullScreenCover for retrim has no fallback when athlete is nil
   File: VideoPlayerView.swift:253-257

   The fullScreenCover content checks if let athlete = clip.athlete,
   but if athlete is nil, it renders nothing (empty view). The user
   sees a blank full-screen cover with no way to dismiss it except
   swiping (which may not work on fullScreenCover). The menu button
   at line 167 guards on clip.athlete != nil before showing the trim
   option... wait, it guards on clip.isUploaded && clip.athlete != nil.
   But it checks clip.athlete, not clip.athlete != nil explicitly:

     if clip.isUploaded && clip.athlete != nil {

   This guard prevents the trim option from appearing when athlete is
   nil, so in practice the fullScreenCover won't be triggered. But if
   showingRetrimFlow were ever set programmatically, it would show a
   blank screen. Minor defensive issue.


8. VideoClipInfoCard hidden on iPad
   File: VideoPlayerView.swift:130

   The info card is hidden when hSizeClass == .regular (iPad). This
   means iPad users never see the play result, game opponent, highlight
   badge, or date below the video. The comment says "hidden in landscape
   so video fills the screen" but the condition also excludes all iPad
   layouts (which are always .regular horizontal).

   Fix: Condition should probably be just vSizeClass == .compact for
   landscape, not also checking hSizeClass:

     if vSizeClass != .compact {
         VideoClipInfoCard(clip: clip)
     }


9. Menu ProgressView in save-to-photos label won't animate
   File: VideoPlayerView.swift:181-186

   The "Save to Photos" menu item shows a ProgressView spinner while
   saving. But Menu items are rendered as static snapshots -- the
   spinner won't actually animate. The user sees a frozen spinner icon.
   The button is also disabled during save, so the menu would need to
   be re-opened to see the state change anyway.

   Fix: Remove the ProgressView branch from the menu label. The
   disabled state is sufficient. Or show progress outside the menu.


MINOR / CLEANUP
===============

10. Toolbar placement uses deprecated string-style API
    File: VideoPlayerView.swift:138, 210

    .navigationBarLeading and .navigationBarTrailing work but are
    older naming. The canonical form is .topBarLeading and
    .topBarTrailing (iOS 14+). Both compile fine.


11. No loading state shown during cloud URL signing
    File: VideoPlayerView.swift:377-386

    When the signed URL fetch fails, the error is logged but the
    code falls through to use the original cloudURL (which is likely
    an unsigned Firebase Storage URL). This may fail or succeed
    depending on security rules. There's no user feedback that the
    signing failed and playback might not work.


12. Error message for missing video is generic
    File: VideoPlayerView.swift:341

    "Video file not found. It may have been moved or deleted." is shown
    even when the issue is that no cloudURL exists for download. A more
    specific message like "Video not available locally or in the cloud"
    would help users understand the situation.


13. Photos import unused if saveToPhotos converted to async
    File: VideoPlayerView.swift:14

    The Photos framework import is used for PHPhotoLibrary. This is
    fine and necessary. Just noting it for completeness.
