BUGS (should fix)
=================

1. startScheduledSession doesn't set startedAt locally
   File: CoachSessionManager.swift:196-201

   When a scheduled session transitions to live, startedAt is set via
   FieldValue.serverTimestamp() on Firestore but the local session object
   never gets a startedAt value. This means LiveSessionCard's timer
   (Text(startedAt, style: .timer) at LiveSessionCard.swift:96) won't
   show because session.startedAt is nil until the next fetch/listener
   update.

   Current code:
     var session = scheduledSessions.remove(at: idx)
     session.status = .live
     activeSession = session
     sessions.insert(session, at: 0)

   Fix: Also set session.startedAt = Date()
   Note: Requires fix #2 first (startedAt is currently let).


2. CoachSession.startedAt is declared as let -- prevents local mutation
   File: FirestoreModels.swift:256

   startedAt being let prevents the manager from setting it locally after
   server timestamp writes. This blocks fix #1 and makes it impossible to
   update startedAt anywhere in the codebase.

   Fix: Change "let startedAt: Date?" to "var startedAt: Date?"


3. ClipReviewSheet.shareClip sends notification to wrong recipient
   File: ClipReviewSheet.swift:284-286

   Code:
     await ActivityNotificationService.shared.postNewVideoNotification(
         ...
         coachIDs: [folder.ownerAthleteID],  // <-- ATHLETE's ID, not coach
         ...
     )

   The parameter is named coachIDs but the athlete's owner ID is passed.
   The notification likely targets the wrong person or silently fails.
   The intent is to notify the athlete that a clip was shared with them.

   Fix: Verify the postNewVideoNotification API and pass the correct
   recipient. If the API expects athlete IDs for this case, the parameter
   name is misleading. If it truly expects coach IDs, this is sending to
   the wrong target entirely.


4. Offline check blocks clip enqueue instead of letting queue retry
   File: DirectCameraRecorderView.swift:445-448

   If the device is momentarily offline, the clip is NOT enqueued at all.
   The user sees an alert with "Try Again" or "Discard" options. But
   UploadQueueManager already handles retries with exponential backoff.
   The clip should be enqueued regardless of connectivity (like the
   athlete flow does), with the queue handling retries.

   Risk: If the user taps "Discard" on the offline alert, the recorded
   clip is permanently lost even though it could have been uploaded later.

   Fix: Remove the connectivity guard. Enqueue the clip unconditionally
   and let UploadQueueManager handle retry logic.


POTENTIAL ISSUES (lower severity)
==================================

5. fullScreenCover uses potentially empty session ID
   File: CoachDashboardView.swift:169

   Code:
     CoachSessionContext(sessionID: session.id ?? "", session: session)

   If session.id is nil, an empty string becomes the session ID. The
   upload path would then reference an invalid Firestore document path.
   Should use guard-let to avoid presenting the camera with bad data.


6. No double-tap guard on SessionAthletePickerOverlay
   File: SessionAthletePickerOverlay.swift:33-35

   When the coach taps an athlete, onSelect fires immediately. The
   overlay doesn't dismiss itself -- it relies on the parent's
   saveCoachClip -> dismiss(). If save/dismiss is slow, a rapid double-
   tap could fire onSelect twice for different athletes, potentially
   uploading the same clip to two different folders.

   Fix: Add a @State isSaving flag, disable buttons while saving.


7. No duplicate clip upload protection in uploadClip
   File: CoachSessionManager.swift:417-470

   No deduplication check exists. If the same clip URL gets passed twice
   (e.g., from a UI race condition), two uploads would be enqueued. The
   copyItem would fail on the second call since the file already exists
   at stablePath, and the error returns early -- silently losing the
   second attempt. If the first attempt also had issues, the clip is
   lost entirely.


8. cleanupAbandonedSessions skips review phase
   File: CoachSessionManager.swift:97-107

   Abandoned sessions jump directly from .live to .completed via
   completeSession(), skipping the .reviewing state. Any clips recorded
   in that session never get a review phase. This may be intentional for
   truly abandoned (24hr+) sessions, but coaches lose the review
   opportunity for any clips captured before abandonment.


IMPROVEMENTS (nice to have)
============================

9. fetchSessions makes 4 sequential Firestore queries
   File: CoachSessionManager.swift:260-274

   The four fetchSessionsWithStatus calls are awaited sequentially. They
   could be parallelized with async let for faster dashboard load:

     async let live = fetchSessionsWithStatus(coachID: coachID, status: .live, limit: 10)
     async let reviewing = fetchSessionsWithStatus(coachID: coachID, status: .reviewing, limit: 20)
     async let completed = fetchSessionsWithStatus(...)
     async let scheduled = fetchSessionsWithStatus(...)
     let (liveResult, reviewingResult, completedResult, scheduledResult) =
         try await (live, reviewing, completed, scheduled)


10. CoachSession equality only checks id
    File: FirestoreModels.swift:247

    The == operator only compares id, so SwiftUI won't detect changes to
    status, clipCount, notes, etc. for onChange or ForEach diffing. Since
    the manager is @Observable this mostly works at the array level, but
    any view storing a local copy of a session won't notice field changes.


11. Real-time listener only covers .live sessions
    File: CoachSessionManager.swift:369-404

    The snapshot listener only watches status == "live". If a session
    transitions to .reviewing externally (e.g., from another device), the
    listener won't pick it up. The comment in the code acknowledges this,
    but for multi-device coaches it means the reviewing state is delayed
    until the next fetchSessions call (pull-to-refresh or app foreground).


12. StartSessionSheet doesn't pass onInviteAthlete callback
    File: CoachDashboardView.swift:143

    The sheet is created as StartSessionSheet() with no onInviteAthlete
    closure. When there are no athletes available, the "Invite an
    Athlete" button in the empty state won't appear (it checks
    onInviteAthlete != nil at StartSessionSheet.swift:164). The dashboard
    already has the showingInviteAthlete state -- it could be wired up:

      StartSessionSheet(onInviteAthlete: { showingInviteAthlete = true })
