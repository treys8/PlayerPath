CRITICAL / BUG
==========================================================================

1. [Bug] DashboardView — @StateObject init runs on every parent re-render
   File: DashboardView.swift:47
   `@StateObject(wrappedValue: GamesDashboardViewModel(athlete:modelContext:))`
   in the init is correct for the first creation, but if the parent
   rebuilds DashboardView with a DIFFERENT athlete (e.g., after athlete
   switch), @StateObject does NOT recreate — it keeps the old ViewModel
   bound to the previous athlete. The dashboard will show stale data for
   the wrong athlete until `viewModel.refresh()` fires.
   FIX: Either use `.id(athlete.id)` on DashboardView from the parent
   so SwiftUI tears down and recreates it, or use @State + .task(id:)
   pattern to reinitialize when athlete changes.

2. [Bug] DashboardView — Quick Actions only shows when hasLiveGame is true
   File: DashboardView.swift:292
   The `quickActionsSection` is gated on `hasLiveGame`. This means the
   "New Game" and "Record Live" buttons ONLY appear when a game is already
   live. Users with no live game see no quick actions at all. The "New
   Game" button should arguably always be visible, and "Record Live"
   should be gated on hasLiveGame.
   SEVERITY: UX issue, not a crash bug. But the intent seems wrong — new
   users with no live game get no quick actions on their dashboard.

3. [Bug] DashboardView — refreshable sync ignores athlete context
   File: DashboardView.swift:225-233
   `SyncCoordinator.shared.syncAll(for: user)` syncs all data for the
   user. This is correct. But the check `if user.firebaseAuthUid != nil`
   uses a property that may be nil for locally-created users who haven't
   finished onboarding. In that case, pull-to-refresh only calls
   `viewModel.refresh()` (local data reload) with no cloud sync —
   silently degraded behavior with no indication to the user.
   SEVERITY: Low — only affects edge case where user exists locally
   without Firebase auth.

4. [Bug] DashboardNextStepCard — FetchDescriptor in computed property on every body eval
   File: DashboardNextStepCard.swift:65
   `tipsEnabled` does `try? modelContext.fetch(FetchDescriptor<UserPreferences>())`
   inside a computed property that's evaluated on every body render.
   This is a synchronous database query in the render path. Every time
   the dashboard re-renders, this fetches from SwiftData.
   FIX: Cache the value in @State and update via .task or .onChange.

POTENTIAL ISSUE
==========================================================================

5. [Potential] DashboardView — athlete switch via NotificationCenter is fragile
   File: DashboardView.swift:115-118
   Athlete switching is done via `NotificationCenter.default.post(name:
   .switchAthlete, object: ath)`. This relies on someone upstream
   observing this notification and updating binding state. If the observer
   isn't registered (e.g., view hierarchy changes), the switch silently
   fails. Consider using a shared @Observable coordinator or environment
   binding instead.

6. [Potential] DashboardView — multiple heavy computed properties on athlete
   File: DashboardView.swift:356-364
   `(athlete.seasons ?? []).count`, `(athlete.practices ?? []).count`,
   `(athlete.photos ?? []).count` are all evaluated during body. Each
   accesses optional relationship arrays that SwiftData may fault in
   lazily. With many items, this could cause UI hitches during scroll.
   Consider caching these counts alongside `updateCachedStats()`.

7. [Potential] DashboardView — createNewGame posts two notifications in sequence
   File: DashboardView.swift:414-426
   `createNewGame()` posts `postSwitchTab(.games)` then immediately posts
   `.presentAddGame`. The receiving views may not have appeared yet from
   the tab switch, so the second notification could be lost.
   The Task { @MainActor in } wrapper doesn't add any delay — both posts
   happen in the same run loop iteration.
   FIX: Post `.presentAddGame` after a brief delay, or use a state-driven
   approach (e.g., NavigationCoordinator pending action).

8. [Potential] LiveGameCard — isPulsing animation restarts on every scenePhase change
   File: LiveGameCard.swift:45-51
   Setting `isPulsing = true` on every `.active` scene phase change
   restarts the animation. The `.repeatForever` animation is already
   running, so re-triggering it may cause a visual glitch (brief pause
   then restart).
   MINOR: Cosmetic only.

9. [Potential] DashboardStatCard — isAnimating never resets
   File: DashboardStatCard.swift:67-70
   `isAnimating` is set to true on `onAppear` and never reset. If the
   view disappears and reappears (e.g., tab switch), the entrance
   animation won't replay because `isAnimating` is still true from @State
   persistence. Same issue in DashboardFeatureCard.swift:99-103.
   This is intentional for most cases (avoid repeated animations), but
   worth noting if the design intent is to animate on every appearance.

MINOR / IMPROVEMENT
==========================================================================

10. [Minor] DashboardView — dead @State pulseAnimation initialization
    File: DashboardView.swift:48
    `self._pulseAnimation = State(initialValue: false)` in init is
    redundant — it matches the default at line 23. Remove the init line.

11. [Minor] DashboardFeatureCard — unused isPressed state
    File: DashboardFeatureCard.swift:18
    `@State private var isPressed = false` is used only by the custom
    `FeatureCardButtonStyle`. The binding bridge works but is an unusual
    pattern. The `PressableCardButtonStyle` used elsewhere achieves the
    same effect without a @Binding. Consider unifying the approach.

12. [Minor] DashboardView — formatBattingAverage edge case
    File: DashboardView.swift:442-448
    For values like 0.9995 (rounds to 1.000), the function takes the
    `value >= 1.0` path and returns "1.000" — correct. But for 0.9994
    which rounds to 999 thousandths, it returns ".999" which is also
    correct. Just noting the boundary is handled properly.

13. [Improvement] DashboardView — Quick Stats section always shows batting stats
    The Quick Stats section always shows AVG, SLG, and Hits. For pitchers,
    these stats may be empty/zero and the section provides no value.
    Consider conditionally showing pitching stats (ERA, K, etc.) based on
    athlete position/role or which stats have data.

14. [Improvement] DashboardNextStepCard — no animation on dismiss
    File: DashboardNextStepCard.swift:114-118
    The dismiss button calls `withAnimation { ... }` but the card has no
    explicit transition. The default transition (fade) works but a
    `.transition(.move(edge: .top).combined(with: .opacity))` would be
    smoother.

15. [Improvement] DashboardView — consider using @Query for counts instead of accessing relationships
    Instead of `(athlete.games ?? []).isEmpty` which loads the full
    relationship, a `@Query` with a count predicate would be more
    efficient for display purposes only.

==========================================================================
SUMMARY
==========================================================================

Critical bugs to fix first:
  #1 — @StateObject not recreated when athlete changes. Dashboard shows
       wrong athlete's data after switch until refresh fires.
  #2 — Quick Actions section hidden when no live game, meaning new users
       see no quick actions on their dashboard.
  #4 — FetchDescriptor called synchronously on every body render in
       DashboardNextStepCard.

The Dashboard is well-structured overall with good accessibility support
(labels, hints, reduce-motion checks), proper skeleton loading states,
and clean component extraction. The main architectural issue is the
@StateObject lifecycle (#1) which can show stale data after athlete
switching. The notification-based navigation pattern (#5, #7) is used
extensively and works but is inherently fragile compared to state-driven
navigation.
