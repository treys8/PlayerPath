HIGHLIGHTS AUDIT — HighlightsView, HighlightsViewModel
======================================================

## HighlightsView.swift

BUG #1 — batchUploadSelected bypasses UploadQueueManager (lines 560-608)
  Uploads are done inline via VideoCloudManager.shared.uploadVideo() in a sequential loop,
  bypassing the UploadQueueManager entirely. This means:
  - No exponential backoff retry on failure (only a single attempt per clip)
  - No background upload support
  - No deduplication with clips already queued by UploadQueueManager
  - Storage quota (cloudStorageUsedBytes) is updated manually here but may differ from how
    UploadQueueManager calculates it, causing drift
  All other upload paths in the app use UploadQueueManager — this should too.

BUG #2 — batchUploadSelected doesn't check storage quota before uploading (lines 560-608)
  The upload loop starts without checking if the user has enough cloud storage remaining.
  It only updates cloudStorageUsedBytes after each successful upload. If the user is near
  their quota limit, they could exceed it mid-batch with no warning.

BUG #3 — shareClip uses UIKit presenter directly (lines 523-536)
  Uses UIApplication.shared.connectedScenes to get rootViewController and present
  UIActivityViewController. On iPad, popoverPresentationController.sourceView is set to
  rootVC.view (the entire screen) with no sourceRect — the popover will anchor to the
  top-left corner. Same issue in batchShareSelected (lines 610-635). Also, if a sheet or
  fullScreenCover is already presented, rootViewController.present will silently fail.

BUG #4 — recomputeTask not cancelled on view disappear (line 32)
  The debounced Task stored in recomputeTask is never cancelled when the view disappears.
  If the user navigates away quickly, the task can fire and mutate viewModel after the
  view is gone. Should cancel in .onDisappear.

BUG #5 — SeasonFilterMenu shows "no season" option based on filtered highlights (line 303)
  `viewModel.highlights.contains(where: { $0.season == nil })` only checks the currently
  displayed (paginated) highlights, not all highlights. If the only no-season clips are
  beyond the display limit, the "No Season" filter option won't appear.

MINOR #6 — displayLimit resets to 50 on every refilter (HighlightsViewModel line 99)
  Same issue as PracticesViewModel — if user has loaded 200 highlights via pagination,
  any filter/search/sort change resets them to 50 and they lose scroll position.

MINOR #7 — Memoization hash only checks first 20 clips (HighlightsViewModel lines 122-128)
  recomputeGroupedHighlights uses a hash of clips.count + first 20 clip IDs for memoization.
  If clips beyond index 20 change (e.g., a clip is deleted at index 25), the hash won't
  change and the grouping won't update. Should hash all clip IDs or use a different
  invalidation strategy.

MINOR #8 — groupedHighlights computed but never used in the view
  The view renders a flat grid via `viewModel.highlights` (line 216). The grouped highlights
  (`viewModel.groupedHighlights`) are computed on every refilter via recomputeGroups() but
  never referenced in the view body. This is wasted computation. Either the grouped view was
  removed and the grouping code should be cleaned up, or the grouping was intended but not
  wired in.

MINOR #9 — Practice filter logic uses clip.game == nil (HighlightsViewModel line 77)
  Filtering for "practice" highlights checks `clip.game == nil`. But a practice clip should
  have clip.practice != nil. A clip with game == nil and practice == nil (orphaned clip)
  would incorrectly show up under the "Practice" filter.

MINOR #10 — Migration runs on every .task invocation (lines 109-142)
  migrateHitVideosToHighlights() is called in .task which runs every time the view appears.
  While the hasCompletedMigration guard prevents re-processing, it still iterates all videos
  to check the guard on each appear. Minor perf hit for large video libraries.

MINOR #11 — No user feedback on share/upload failures
  shareClip silently returns if file doesn't exist (line 524-526). batchShareSelected
  silently returns if no files exist (line 620-622). batchUploadSelected logs errors but
  never shows the user a toast or alert about failed uploads. The user gets a haptic error
  but no explanation.


## HighlightsViewModel.swift

MINOR #12 — updateAvailableSeasons uses Set on reference types (line 187)
  Same issue as PracticesViewModel — Set(seasons) relies on Season's Hashable conformance.
  Fragile if Season doesn't hash by id.

MINOR #13 — onItemAppear pagination threshold is hardcoded to 10 (line 114)
  With displayLimit at 50, the next page loads when the user reaches item 40. For fast
  scrollers or large grid layouts (iPad), this may not trigger early enough, causing a
  visible gap before new items load.


## CRITICAL SUMMARY

1. BUG #1 — Batch upload bypasses UploadQueueManager (no retry, no background, quota drift)
2. BUG #3 — Share popover misanchored on iPad, fails if sheet already presented
3. BUG #8 (MINOR) — groupedHighlights computed but never rendered (dead code / wasted work)
