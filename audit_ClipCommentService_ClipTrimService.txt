[CC1] MEDIUM — ClipComment.id is optional Identifiable — same pattern
  as QuickCue/DrillCard
  Line: 26
  `var id: String?` while conforming to `Identifiable`. During the
  `compactMap` decode loop (lines 118-127, 154-162), `id` is set from
  `doc.documentID` AFTER the `data(as:)` decode, so a freshly-decoded
  comment momentarily has `id = nil` before the next line sets it. If
  Codable's `data(as:)` ever fails partially, the id assignment is
  skipped. In practice the decode either fully succeeds or fully fails
  (compactMap skips nil), so this isn't currently exploitable. But the
  optional id means ClipComment can't be used in Sets or as Dict keys.
  Fix: Decode `id` from `doc.documentID` as part of initialization,
  or use a non-optional `id` with a default.

[CC2] MEDIUM — Cache never expires or bounds its size
  Lines: 53, 129, 165
  `cache: [String: [ClipComment]]` grows unboundedly — one entry per
  unique clipId ever fetched. Each entry holds up to 100 comments.
  In a long coach session reviewing many clips, this accumulates.
  There's no TTL or LRU eviction.
  Fix: Add a simple bounded cache (e.g., max 50 entries, evict oldest)
  or clear cache entries when the listener detaches.

[CC3] MEDIUM — Listener and fetch can race
  File: ClipCommentSection.swift, lines 82-89
  The view seeds comments via `fetchComments` (one-shot), then attaches
  a snapshot listener via `attachListener`. If a new comment arrives
  between the fetch response and the listener attachment, it's missed.
  The listener's first snapshot includes the full state, so the next
  listener callback would cover it — but there's a brief window of
  stale data.
  Fix: Attach the listener first (which fires immediately with current
  state), and skip the initial fetchComments. Or use the listener's
  first callback as the seed.

[CC4] LOW — No delete comment functionality
  The service has postComment and fetchComments but no deleteComment.
  Comments once posted can't be removed by the author. This may be
  intentional for v1 ("coach replies, athlete reads" per the header).
  Suggestion: Add deleteComment if coaches should be able to remove
  their own comments, or document that deletion isn't supported.

[CC5] LOW — authorRole is a raw String instead of UserRole enum
  Line: 29
  `let authorRole: String // "athlete" | "coach"` — the app has a
  `UserRole` enum in FirestoreModels.swift. Using String means any
  typo would create an unrecognized role.
  Suggestion: Use `UserRole` for type safety. The Codable rawValue
  mapping would handle Firestore serialization automatically.

[CC6] LOW — Duplicate annotationCategory computed property
  Lines: 36-39
  Third instance of the identical `annotationCategory` pattern (also on
  QuickCue in AnnotationModels.swift:59 and VideoAnnotation in
  FirestoreModels.swift:195). Flagged in audit_AnnotationModels.txt [AM6].
  Suggestion: Share via protocol extension.

[CC7] LOW — limit(to: 100) is hardcoded in both fetch and listener
  Lines: 115, 147
  A clip with 101+ comments silently drops the excess. No UI indication
  that older comments exist.
  Suggestion: Acceptable limit for v1, but consider pagination or a
  "load more" affordance if comment volumes grow.

[CT1] MEDIUM — Quota rollback on upload failure doesn't save context
  Lines: 184, 192
  Before upload, `user.cloudStorageUsedBytes` is decremented (line 184).
  On upload failure, it's incremented back (line 192). But neither line
  is followed by `context.save()`. If the app crashes after line 184
  but before line 196 (post-upload save), the quota in SwiftData is
  decremented without being restored. The next launch would show
  artificially low usage.
  Fix: Either wrap the quota decrement + upload in a transaction pattern
  that saves before and after, or accept the risk (SyncCoordinator may
  reconcile quota from Firestore on next sync).

[CT2] MEDIUM — Three context.save() calls in one method
  Lines: 173, 207, 229
  `applyTrim` calls `context.save()` three times:
    1. After local metadata update (line 173)
    2. After upload success + cloudURL update (line 207)
    3. After Firestore sync success (line 229)
  Each save is intentional (the comments explain why), but if save #2
  succeeds and save #3 fails (TrimError.saveFailed), the clip is left
  with `needsSync = true` even though Firestore was actually updated.
  This is benign (next sync would no-op) but wastes a sync cycle.
  Fix: Acceptable — the multi-save design is intentional for crash
  resilience. The stale `needsSync` flag is self-healing.

[CT3] MEDIUM — replaceItemAt runs on @MainActor
  Line: 141
  `FileManager.default.replaceItemAt(oldLocalURL, withItemAt:
  trimmedSourceURL)` is a synchronous file I/O operation running on
  the main actor. For large video files, this atomic replace involves
  copying data on disk, which can block the UI for 50-200ms.
  Fix: Offload to a background thread:
  `try await Task.detached { try FileManager.default.replaceItemAt(...) }.value`

[CT4] LOW — Old thumbnail not cleaned up on re-trim
  Lines: 149-164
  `generateThumbnail` creates a new thumbnail file. Since the fileName
  is stable, it overwrites the same path. But if the path generation
  logic ever changes (e.g., hash-based naming), the old thumbnail file
  would be orphaned. The code correctly invalidates the ThumbnailCache,
  but doesn't explicitly delete the old thumbnail file.
  Suggestion: Acceptable — current naming is deterministic from fileName.

[CT5] LOW — TrimError uses user-facing messages in errorDescription
  Lines: 33-58
  Each TrimError case returns a user-friendly string. This is good for
  alert presentation but means error descriptions can't be used for
  logging without mixing user-facing text into logs.
  Suggestion: Minor — the calling view (RetrimSavedClipFlow) shows
  these directly in alerts, which is the intended use.

[CT6] LOW — Precondition checks clip.isUploaded but not the file
  existence
  Lines: 92-98
  `applyTrim` checks `clip.isUploaded` and `clip.cloudURL != nil`
  but doesn't verify that the local file at `clip.resolvedFilePath`
  actually exists. If the local file was deleted (e.g., by iOS storage
  pressure), `replaceItemAt` would fail with a confusing error.
  Suggestion: Add a `FileManager.fileExists(atPath:)` check before
  the replace, with a clear error message.

Notes:
  - ClipCommentService is clean and well-structured. The singleton +
    @MainActor pattern provides thread safety. The snapshot listener
    integration is correct — detached on disappear, cache updated
    from listener.
  - ClipTrimService is impressively thorough. The multi-phase save
    strategy (save after local changes, save after upload, save after
    sync) is deliberately designed for crash resilience. The quota
    management with rollback on failure shows careful thinking.
  - Both services properly use structured error types. ClipTrimService's
    TrimError is particularly well-designed with user-facing messages.
  - ClipTrimService correctly calls `invalidateResolvedPathCache()`
    (line 147) — the only call site that does this properly (noted in
    audit_VideoClipModel.txt, now moot since updateFilePath exists).
